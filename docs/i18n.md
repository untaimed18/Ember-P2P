# Internationalization (i18n)

Ember uses [Paraglide JS](https://github.com/opral/paraglide-js)
v2 in the SvelteKit frontend. Translations live as plain JSON
files under `/messages`, get compiled at build time into typed
TypeScript message functions under `src/lib/paraglide/`, and are
imported into components as `import * as m from
'$lib/paraglide/messages'`.

This document is the operational handbook for everyone touching
strings in the UI: how to add a new key, how to migrate an
existing component, how plurals/interpolation work, and how the
language switcher resolves the active locale.

---

## Quick reference

- **Source of truth:** `messages/en.json` (base locale).
- **Translations:** `messages/<locale>.json` (currently `es.json`).
- **Compiled output:** `src/lib/paraglide/` (auto-generated, do
  not edit, do not delete; checked in so a fresh clone builds
  without running the Vite plugin first).
- **Project config:** `project.inlang/settings.json`.
- **Vite wiring:** `vite.config.js` — the `paraglideVitePlugin`
  re-emits the compiled output on each build / HMR update.
- **Runtime helper:** `src/lib/i18n.ts` — re-exports
  `getLocale`/`setLocale`/`locales`, plus the
  `applyDocumentLang()` and `translateErrorCode()` helpers used
  by the layout and Tauri-error path.

## Resolution strategy

The Vite plugin is configured with:

```js
strategy: ["localStorage", "preferredLanguage", "baseLocale"]
```

Paraglide picks the first strategy that yields a usable locale:

1. **`localStorage`** — `localStorage["PARAGLIDE_LOCALE"]`.
   Written by `setLocale()` whenever the user changes language
   in Settings.
2. **`preferredLanguage`** — `navigator.language`. Used on first
   launch when the user has never explicitly chosen.
3. **`baseLocale`** — falls back to `"en"` if neither of the
   above produces a locale that's actually compiled in.

A locale change calls `setLocale(next)` (from
`$lib/paraglide/runtime`), which writes to localStorage and
triggers a full window reload by default. The reload is
intentional: it guarantees every cached `m.*()` call site
re-renders with the new strings, with no need to thread Svelte
reactivity through 30+ components.

`<html lang>` is set by `applyDocumentLang()` in
`src/routes/+layout.svelte` so screen readers, browser
spellcheck, and `:lang(xx)` CSS selectors all behave correctly.

## Migrating a string in three steps

1. **Add a key to `messages/en.json`**, then to every other
   `messages/<locale>.json`.
2. **Replace the literal in your component:**

   ```svelte
   <script lang="ts">
     import * as m from '$lib/paraglide/messages';
   </script>
   <button>{m.common_cancel()}</button>
   ```

3. **Rebuild.** Paraglide regenerates `src/lib/paraglide/` on
   the next `npm run dev` / `npm run build`. New keys show up
   immediately in autocomplete; missing keys are TS errors.

## Naming convention

Keys are flat, snake_case, scoped by surface:

```
common_*          shared across the whole app
nav_*             top-level navigation labels
sidebar_*         primary sidebar
statusbar_*       bottom status bar
chat_dock_*       multi-chat dock
chat_*            single chat conversation
toast_*           toast container
confirm_*         ConfirmDialog
shortcuts_*       KeyboardShortcutsDialog
about_dialog_*    AboutDialog
close_dialog_*    CloseAppDialog
splash_*          SplashScreen
settings_*        Settings page
error_*           machine-stable backend error codes
network_status_*  KAD/ED2K status mapping
language_name_*   language picker labels
```

When adding a new route, prefix every key with the route name
(`friends_*`, `transfers_*`, etc.). Keep the prefix short — keys
are typed identifiers, not English sentences.

## Interpolation

Use named placeholders. Paraglide infers the params type from
the catalog:

```json
"settings_language_system": "System ({system})"
```

```ts
m.settings_language_system({ system: "es-ES" })
// → "System (es-ES)"
```

Placeholders must match across every locale file or the build
fails. Spanish-only changes should keep the same `{name}` slots
as English, even if word order differs.

## Pluralization

The `plugin-message-format` we use does NOT support inline ICU
plurals (`{count, plural, one {…} other {…}}`). Instead, declare
two keys and pick between them in code:

```json
"sidebar_chats_unread_title_one":   "1 unread message",
"sidebar_chats_unread_title_other": "{count} unread messages"
```

```ts
const title = unread === 1
  ? m.sidebar_chats_unread_title_one()
  : m.sidebar_chats_unread_title_other({ count: unread });
```

For multi-axis plurals (e.g. peers AND sources in StatusBar's
EPX tooltip), use a 4-key `_one_one`, `_one_other`,
`_other_one`, `_other_other` matrix. It's verbose but lets
translators control word order in each cell, which matters for
languages where the grammatical agreement of "1 source" vs
"N sources" differs.

This is the only spot Paraglide is meaningfully less ergonomic
than svelte-i18n / i18next. The tradeoff bought us tree-shaking
and type safety, which we want everywhere else.

## Backend errors

Tauri commands that surface user-facing errors should emit
**stable string codes** (e.g. `"FriendNotFound"`,
`"InvalidHash"`) rather than free-form English. The frontend
catches them and runs them through `translateErrorCode(e)` in
`src/lib/i18n.ts`, which maps known codes to translated
messages and falls through to the raw string for anything
unmapped.

To add a new backend error:

1. Have the Rust side return the code as the exact error
   message (no surrounding text):
   `bail!("FriendNotFound")` rather than
   `bail!("Could not find friend {hash}")`.
2. Add an `error_<snake>` key to every locale file.
3. Add a `case` to `translateErrorCode` in
   `src/lib/i18n.ts`.

Anything not yet codified keeps showing the English server
message — adding new codes is non-breaking.

## Component migration checklist

Pick a component (route or shared), then:

- [ ] `import * as m from '$lib/paraglide/messages';`
- [ ] Walk the template top-to-bottom and replace every
      user-visible literal: button text, headings, paragraphs,
      `placeholder=`, `aria-label=`, `title=`, `alt=`, error
      messages set in the script block, validation strings.
- [ ] Skip developer-facing strings: `console.log/warn/error`
      messages, internal exceptions never shown to the user,
      log lines.
- [ ] Re-run `npm run build` once. Paraglide compiles only on
      build; new keys aren't visible to TS until then.
- [ ] Run `npm run check` — missing keys become TS errors.
- [ ] Translate every new key in `messages/es.json`. Empty
      strings are not allowed; use the English text as a
      placeholder if you genuinely don't have a translation
      yet, and tag it with a TODO comment in the JSON file.

## Migrating a whole route

The recommended order: pick the highest-traffic route first
(`/transfers` or `/library`), then work outward:

1. Read the route page top to bottom.
2. Open `messages/en.json` and append a section under the
   appropriate prefix.
3. Migrate the template + script.
4. Translate to Spanish.
5. Smoke-test in the browser by toggling locale via
   Settings → Language.

## Adding a new locale

1. Append the locale code to `project.inlang/settings.json` →
   `locales`.
2. Copy `messages/en.json` to `messages/<locale>.json` and
   translate.
3. Add a `language_name_<locale>` key to every locale file (the
   name in that language).
4. Add a `case '<locale>'` in `languageLabel()` in
   `src/lib/i18n.ts`.
5. Rebuild. The Settings picker auto-renders the new option
   from `locales`.

## Inlang tooling

Optional but recommended for translators:

- **VS Code extension:** "Sherlock — Inlang" surfaces
  translations inline in your editor.
- **Web editor:** [fink.inlang.com](https://fink.inlang.com)
  works against the project's git repo without a local
  toolchain.

Neither is required to ship — both message files are plain JSON
that any text editor will handle.
